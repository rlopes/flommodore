# Flommodore — Implementation Plan

**v1.1 — incorporates the Block 0 specification amendments and the design-audit plan
improvements (P1–P8). Supersedes the v1.0 plan.**

## Overview

This document breaks the Flommodore emulator and toolchain implementation into small,
concrete, independently completable tasks. Each task ends with something that compiles
and runs. Tasks are ordered so that each one builds on the last.

---

## Implementation Principles

1. **Always have a running program** — every task ends with something that compiles and
   runs, even if it does nothing useful yet
2. **Bottom up** — infrastructure before features (bus before CPU, CPU before video)
3. **Test as you go** — each component gets its test ROM before the next component starts.
   Test ROMs are generated by `zig build genroms` using the shared `encode.zig` instruction
   encoder (Block 2) — **one source of encoding truth** shared with the assembler
4. **Deterministic by default** — fixed LFSR/PRNG seeds, headless mode, and golden-output
   tests make "looks right" into regression protection

---

## Block 0 — Specification Corrections ✓ COMPLETE

Resolved before any code: vector width (4 bytes at `$FFFC0`), 20-bit register model and
`LI`/`LUI` semantics, fixed shadow window (`$3C000`), canonical load address (`$04100`)
and `.flapp` v1.1 header, software IRQ dispatch, complete opcode assignment table,
`MFSR`/`MTSR`, SSP and the 8-byte interrupt frame, per-op flag table, 14.4 MHz clock,
1 cycle/instruction, scanline timing tables, 5bpp removal, AUR-1 numeric model, and the
toolchain format fixes. See *flommodore-spec-amendment-v1.1.md* (LOCKED) and the v1.1
phase documents.

### Bus routing

```
$00000–$7FFFF  → RAM
$80000–$80FFF  → I/O (16-bit register per address — Phase 5 access model)
$81000–$FBFFF  → open bus (reads $0000, writes ignored)
$FC000–$FFFFF  → ROM (or the $3C000–$3FFFF RAM window if shadow enabled)
```

### Instruction format reminder

```
Format R:  OPCODE(6) | RD(4) | RA(4) | RB(4) | FUNC(9)=0 | FLAGS(5)=0
Format I:  OPCODE(6) | RD(4) | RA(4) | IMM18(18)
Format J:  OPCODE(6) | ADDR26(26)
```

### Render pipeline (per scanline, driven by the main loop)

```
Main loop runs CYCLES_PER_LINE CPU cycles, then for line Y:
  1. Fire raster IRQ if VSCAN matches Y
  2. Fill scanline with VBGCOL
  3. Bitmap layer (if enabled)
  4. Tile layer (if enabled)
  5. Sprite layer (max 8 per scanline, priority + collision)
  6. Write to SDL3 texture row
After the last visible line: VBLANK IRQ → VSWAP → present
```

### Test protocol

All test ROMs write `$FF` to address `$00010` on pass, any other value on failure.
The headless harness runs for a fixed cycle count (`--max-cycles`) and reads `$00010`.
Video and audio tests additionally compare output hashes against checked-in goldens
(deterministic: fixed LFSR seeds, headless mode).

```
$00010 == $FF   → PASS
$00010 == $00   → not yet complete (still running)
$00010 == other → FAIL (value encodes which assertion failed)
```

## Future Phase — FL Language

The FL higher-level language is not part of this implementation plan. It requires a
dedicated design phase before implementation can begin, covering:

- Type system (primitives, pointers, structs)
- Memory model (stack, heap, arenas)
- Calling convention (must match Gab-16 ABI exactly)
- Compiler architecture (direct-to-assembly or IR)
- Error model
- Standard library scope

FL compiles to `.asm` and feeds the existing assembler → linker pipeline unchanged.
